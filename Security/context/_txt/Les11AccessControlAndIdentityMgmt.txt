Identity Management

Identity Management
· Identity management system (IMS) is a system to create, use, and revoke user identities · can be tricky for large organizations and complex systems · various security requirements can apply depending on the use case: · privacy/anonymity/full transparency · strong link to physical identity · revocability · unforgeability · and others.

User vs. Identity vs. Subject
Identity-based authentication & authorization link 3 entities:
· User is a physical person (or a machine/an agent) · You
· (Internal) Identity is the entry in an IMS to which a user can be associated · Your group for appsec
· Subject is an entity in a system that can make actions/execute programs (needs privileges) · A binary or a shell script that you run

User Access Management

Access Control
· Authentication ensures user identity · Authorization makes sure a subject acting with this identity can only access
(has privileges) resources (objects) allowed for this identity · Audit reviews the records of the system to check compliance with the policy
· Access control policy = "Who is allowed to access what?" · what types of access are permitted · under what circumstances · by whom

Access Control Policies

Access Control Policies
· Different classes of access control policies (schemes): · who sets and who controls the policy · access control rules are based on various groupings of identities
· Examples (not mutually exclusive): · Discretionary Access Control (DAC) · Mandatory Access Control (MAC) · Non-Discretionary Access Control · Role-based access control (RBAC) · Attribute-based access control (ABAC) ·...

Discretionary Access Control (DAC)
· Principle: owner of the resource defines who can access the resource · Typically implemented via ACLs · Users can belong to groups · Access rights are granted if the user has the right privilege or belongs to a group with the right privilege · User that has access rights can transfer these rights to other users (can be done indirectly)
· Examples: controls on social networks; file system access on UNIX systems (Android, Mac, etc.) · Issues: easier to make mistakes; assumes all resources have owners

Mandatory Access Control (MAC)
· Principle: system-wide policy determines who can access a resource based on a sensitivity label · each resource and user have a sensitivity label · if they match, the user can access the resource · access rights cannot be transferred
· Examples: military security clearance systems, organization-wide policies · Issues: more difficult to implement; quite rigid

Non-Discretionary Access Control (NAC)
· Principle: admin defines who can access the resource · Typically implemented via ACLs · Users can belong to groups · Access rights are granted if the user has the right privilege or belongs to a group with the right privilege · User that has access rights cannot transfer these rights to other users (can be done indirectly)

Role-Based Access Control (RBAC)
· DAC: users have access to objects · RBAC: users have roles that have access to objects · Principle: access control is based on roles that users assume in a system.
· access is granted per role · users can have several roles · there can be role hierarchies and constraints
· Example: Brightspace (instructors can access hidden course parts, but cannot submit assignments; students cannot access hidden course parts, but can submit assignments)

ACLs vs. Capabilities
· An ACL (access control list) defines access rights per resource (file, directory) · simple to implement; harder to check access rights at runtime
· A capability defines access rights per user · runtime checks are easier, delegation is easy to implement; harder to modify access rights per a file.
· Both become too cumbersome to maintain in large and dynamic systems

Access Control Privileges in Computer Systems

Subject Classes
· Classes group subjects into categories:
· Owner: the main entity that owns a resource (a creator or an administrator)
· Group: membership of a subject in a group is sufficient to exercise some access rights [A user may belong to multiple groups]
· World: least number of privileges that are granted to users who can access the system itself, but are not the owner and do not belong in a group
· Separate category: superuser

Separation of Duties Constraints

Separation of Duties (SoD)

SoD Conflicts and Constraints

SoD Constraints
· Are used to limit opportunities for insider attacks · Constraint examples:
· A user may not have simultaneously roles r1 and r2 (ever or during a session)
· A user in role r1 must also have role r2 · At most (at least) k users must have role r1
· four eyes principle; two signatures principle
· Examples: At least 2 professors evaluate a bachelor thesis; a student cannot be a student assistant in a course they take.

Multi-level Security Policies

The Bell-La Padula Model
· Principle: Information cannot flow downwards · Focuses on confidentiality · Two properties are enforced:
· The simple security property: no subject may read data from a higher level (no read up)
· The *-property: no subject may write data to a lower level (no write down)
· The Bell-La Padula multi-level security model is basically a MAC · security policy is enforced system-wide

The Biba Model
· Principle: data may be visible to many users, but can only be modified by a user with the right privileges
· Focuses on integrity · Enforced with two properties:
· Simple integrity: a subject may only read data at an equal or higher level (no read down)
· Star Integrity: a subject may only write data at an equal or lower level (no write up)
· Example: everybody can read a website, but only admins can update it.

The Chinese Wall Model
· Principle: data may be visible to many users, only as long as no conflicts of interests arise
· Focuses on resolving CoIs
· Enforced with two properties:
· Simple security: a subject s can be granted access to an object o only if the object is in the same company datasets as the objects already accessed by s (within the Wall), or belongs to an entirely different CoI class.
· *-property: Write access is only permitted if access is permitted by the simple security rule, and no object can be read from a different company dataset than the one for which write access is requested, and contains unsanitized information
· Example: In stock exchange and investment house: it prevents traders from representing clients with conflict interest.

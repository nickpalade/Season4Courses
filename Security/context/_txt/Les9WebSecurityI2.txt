Server-Side Data
À We want ACID transactions:
À Atomic: completed entirely or not at all À Consistent: keeping the database in a valid state À Isolated: results of a transaction are only visible when it is
completed À Durable: once a transaction is committed, its effect persists
irrespectively of system failures
À Database Management Systems (DBMS) provide these properties

SQL Injection

SQL Injection Issue
The core problem: mixing code and data
$result = mysql_query("select * from Users where(email=`$emailadres' and password=`$wachtwoord');");
Vulnerabilities arise when the boundary between code and data blurs

SQL Injection Countermeasures
À Input Validation À do not trust that input is of a good form; verify! À just like for buffer overflows
À Make sure that input is data: À Check it and reject unexpected inputs À Sanitize it by making it to conform to a safe format

Sanitization: Blacklisting
À Delete suspicious characters
- ` ; --
À But sometimes these characters are legal
- "5 o'clock"

Sanitization: Escaping
À Replace problematic characters with safe ones
À ` becomes \' À ; becomes \; À - becomes \À \ becomes \\
À There are standard libraries and methods to do this À This is the first line of defense on the server side
À but sometimes we still need those characters in the SQL queries

Checking: Whitelisting
À Check that the input is safe À Reject suspicious input À Given an invalid input, it might be safer to reject it than try
to fix
À "fixing" may cause issues (also security-related ones) À Principle of fail-safe defaults
À But could be difficult to implement for rich inputs

Sanitization: Prepared Statements

Limit the Impact
À Input validation is a must À It is also useful to mitigate the effects of an attack
À Principle: defence in depth
À Approaches:
À Limit server privileges on the database
À limit commands and tables it can access
À Encrypt sensitive data
À decrypt at the server side if needed

Cookies: Pros and Cons
À Cookies are used to
À keep the state as session identifier À personalize the web pages À track users
À ad networks keep their own (third-party) cookies and can
link users across different web domains

Attack on Cookies
À Cookies are used to keep track of authenticated users
À during the first visit the user will authenticate normally
(e.g., via password)
À upon authentication the server will issue a cookie À subsequently, the client will show this cookie as a proof
of authentication
À A stolen cookie can be used to hijack a session

Session Hijacking
À Holder of a session cookie can access the site with the
privileges of an authenticated user
À Session hijacking consequences:
À perform actions as if done by the user À possibly stealing/corrupting sensitive data

Firesheep

Where Cookies can be stolen

Preventing Cookie Theft

Server-Side Defenses
À Set sensitive cookies to be very short-lived À Invalidate the cookie when the user logs-out À Delete the cookie when the session ends À Make cookies different from session to session À Collect separate information and correlate whether the
present interaction is a valid and legitimate request, based on past activity
À monitoring the referrer fields, checking patterns, etc.

Attacks from Other Websites

URLs with Side Effects
À GET requests can have side-effects on the server side
À even though they are not supposed to
http://bank.com/transfer.cgi?amt=9999&to=attacker
À What happens if a user with a valid session cookie clicks
this link?
À probably not something the user wants À but how to make the user to click?

Cross-Site Request Forgery

CSRF
À Target: Attack on users who are authenticated on vulnerable servers À Attack goal: Issue a request to the server via the user's browser
À It will be interpreted by the server as coming from the user
À Technique: Make the user or the user's browser to "click the link"
À e.g., by sending a web page with something like <img src =...>, XMLHttpRequest, or a hidden form
À Sources of vulnerability:
À the server does not check if the request is legit À it is possible to predict a valid request
À Possible impacts:
À data breach (on client or on server); manipulation of the user's account; change of session state

Client-Side Attacks

Dynamic Web Pages
À Web 1.0:
À Static HTML: page unchanged from request to request
À Dynamic HTML: generated by the server using some data sent by the client or
fetched from the DB
À Web 2.0: HTML pages contain programs in JavaScript that will be executed on
the client side

JavaScript
À JavaScript programs are executed in the browser À They can:
À alter web page contents (modifying the DOM) À track user-side events (clicks, mouse movements, etc.) À issue web requests and read replies À access cookies and other browser data À maintain persistent but invisible connections (AJAX)

Security Issues
À Browsers need to confine JavaScript programs:
À a script from attacker.com should not be able to
À modify bank.com webpage or access its cookies À read user's keystrokes while they type in passwords

Same Origin Policy (SOP)
À Browsers apply Same Origin Policy to confine JavaScript
programs
À All web page elements (layout, cookies, events, scripts)
are associated with an origin
À origin is defined by the domain name (bank.com) that
provided the web page (sent in the cookie also)
À scripts can only access elements with the same origin as
themselves

Cross-Site Scripting (XSS) Attacks
À Scenario:
À attacker.com sends a malicious script À tricks the browser to believe it came from bank.com À profit!
À XSS attacks are basically JavaScript injection

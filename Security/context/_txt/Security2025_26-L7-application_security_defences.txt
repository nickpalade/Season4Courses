ASLR

Preventing exploits

Attacker steps Exploit the fact that libc is loaded at well-known address and jump to it.
Defences?
Address Space Layout Randomization (ASLR): place standard libraries and other useful elements randomly in memory
Prevents return oriented programming (and return-to-libc in particular) Prevents guessing return address in "standard" stack smashing
NOP sled of little help if random space too big Currently deployed on all OSs

ASLR caveats

Preventing exploits

Only randomizes start positions, not contents
To be applied to program instructions, compiler needs to support it All major compilers and OSs should support it nowadays
Needs enough randomness in the possible addresses Should not be a problem with 64-bit address space Needs to actually be random Attackers can try to reduce possibilities with information leak (e.g. printf format vulnerabilities) Memory alignment requirements can reduce possibilities

Control Flow Integrity (CFI)
A function consists of basic blocks (pieces of code) A program execution is made of control transfers among basic blocks
Here: only look at control transfer between functions
Connections between basic blocks can be modeled as Control Flow Graph (CFG) Expected behaviour follows CFG

Detecting exploits Control-flow integrity
Control Flow Integrity: compliance with CFG
To check whether execution is compliant: Compute CFG: during compilation or by static analysis of resulting instructions CFG captures expected behaviour
Monitor behaviour during execution: ensure only paths in CFG are followed Direct calls do not need to be monitored (assuming code is immutable) Indirect calls (via saved values, based on registers, function tables, etc) need to be monitored
Depend on data that can be dynamically altered Monitoring part called In-line Reference Monitor (IRM)

CFI caveats

Detecting exploits Control-flow integrity

CFI defeats all attacks that change expected control flow: ROP, code injection, . . .
CFI analysis and instrumentation can be heavy Control flow analysis: very close to halting problem. . . Check on "hot paths" (functions called many times) can eventually slow down program
Still allows control transfer that are compliant mimicry attacks Constraints for complex programs may limit actual label diversity
Does not prevent data leak or data corruption In particular: read overflow, buffer overflow to affect control flow inside a basic block

Memory safety

A programming language is memory-safe if all possible programs written in the language are memory-safe
A program is memory-safe if all possible executions are memory-safe

Memory-safe execution: Temporal safety: only access memory that is defined (active) Memory is undefined if not yet allocated, uninitialized, deallocated (freed) = Only create pointers through standard means: p = malloc(), p = &x, . . .

= Spatial safety ensure that access is to a legal region, temporal safety ensures that content is still valid

Memory-safe languages

Many languages will take care of memory safety for you Python, Java, C#, Rust, . . . Rust explicitly implements "borrow checker": checks temporal safety at compile-time by automatically marking pointers as invalid
= Easiest way to avoid buffer overflow vulnerabilities Most of the time: add checks at runtime
C and C++ are not memory-safe C and C++ are not going away
Although rustacean are trying really hard to make it so :) Possible to write memory-safe C/C++: compiler-added checks for memory violations
Not universally used due to performance penalties Many ways to create a wrong pointer in C

Language-based prevention: Memory safety and Type safety
Type safety
Each object has a type: int, pointer to int, struct, pointer to function, etc Implicit in some languages, but definitely there
Type safety ensures that operations on objects are always compatible with types Injecting shellcode in a buffer and jumping via a valid buffer pointer is technically memory-safe w.r.t. the pointer. . . but not type-safe!
Dynamically-typed languages (e.g. Python) have additional metadata to identify the type of each object
If an operation is not possible for the current type, throw exception Well-defined behaviour and thus type-safe

Costs of safety
Type and memory safety enforcement is expensive computationally Garbage-collected languages avoid temporal violations, but require more memory (and time) than malloc/free Runtime bounds check take time on each access but avoid spatial violations Verifying violations at compile time removes runtime costs but increases compilation costs Generally a good tradeoff  Rust
